# 希望之洲批量录入说明书

适用对象：负责把问卷、Excel 或在线表格中的信徒资料录入 `hope-state-site` 的操作者或新的 Codex 线程。

## 1. 录入入口与权限

1. 用昵称 `无我` 与正确暗语进入“信徒档案”。
2. 导航中会出现“神谕司务”。
3. 进入后，在“管理密钥”填写 Supabase 中配置的 `ADMIN_KEY`。
4. 使用右侧的“批量卷宗录入”，不要把档案资料粘贴到“群体结算”。

`ADMIN_KEY` 不记录在本说明书、前端代码、截图或导入文本中。

## 2. 批量卷宗录入格式

批量窗口**只接受制表符分隔（TSV）文本**。不接受空格分隔、逗号 CSV、无表头文本或“根据内容猜字段”的格式。

第一行必须是以下表头，顺序不可变，且必须正好 8 列：

```text
昵称	暗语	职业	登神分	觐见分	公开短记	私密备注	天赋
```

每名信徒一行，必须正好 8 列：

| 列 | 用途 | 可公开 |
|---|---|---|
| 昵称 | 唯一档案名，用于后续暗语登录和分数结算 | 是 |
| 暗语 | 个人私密登录凭据 | 否 |
| 职业 | 必须匹配网站职业资料库中的标准职业名 | 是 |
| 登神分 | 初始绝对分数，不是加减分；非负整数，最大 `2147483647` | 是 |
| 觐见分 | 初始绝对分数，不是加减分；非负整数，最大 `2147483647` | 是 |
| 公开短记 | 个人宣言或签名 | 是 |
| 私密备注 | 只在本人私密卷宗中显示 | 否 |
| 天赋 | 只在本人私密卷宗中显示 | 否 |

不要在任意单元格中保留换行或制表符，否则会改变列数。天赋可以在同一单元格中填写长文本；建议写成 `S级：...；A级：...；B级：...；C级：...`。导入后可由信徒在个人面板继续编辑长天赋。

### 严格校验

- 缺少表头、表头列数不为 8、表头顺序不对：直接拒绝。
- 任一数据行多列、少列或未检测到制表符：直接报错。
- 职业不在职业资料库、暗语为空、分数非法：逐行报错。
- 同一批内昵称重复：后端拒绝整批。
- 任一行错误时，后端不会写入任何一行，必须全部修正后重新检查。

## 3. 问卷 / Excel 字段转换

当前问卷导出通常包含以下列：

```text
昵称(信仰+昵称)
职业(命途职业)
暗语(个人信息密码，不可公开)
个人宣言(类似签名，他人可看)
私密备注(建议填写谕行词)
天赋
登神之路分数
觐见之梯分数
```

对应关系：

| 问卷列 | 批量导入列 |
|---|---|
| 昵称(信仰+昵称) | 昵称 |
| 暗语 | 暗语 |
| 职业(命途职业) | 职业 |
| 登神之路分数 | 登神分 |
| 觐见之梯分数 | 觐见分 |
| 个人宣言 | 公开短记 |
| 私密备注 | 私密备注 |
| 天赋 | 天赋 |

### 昵称处理规则

1. 如果填写为 `信仰, 昵称`，使用逗号后的昵称。例如 `时间, 阿幻` 导入为 `阿幻`。
2. 如果问卷平台显示昵称与填写昵称不同，以问卷填写的 `信仰, 昵称` 为准。
3. 若格式反写、信仰名不存在或无法判断，保留问卷平台显示昵称，并在导入前人工确认。
4. 昵称是档案唯一键。导入同名档案会更新该档案，不会新增第二人。

## 4. 职业匹配规则

系统会根据标准职业自动写入信仰、命途和基础战斗职阶。因此导入文本只写“职业”，不要手写信仰或命途列。

职业必须与 Supabase 职业资料库中的名称一致。常见清洗示例：

| 问卷原文 | 导入职业 |
|---|---|
| 战士（格斗专家） | 格斗专家 |
| 法师，欲望主宰 | 欲望主宰 |
| 遗忘医生 牧师 | 遗忘医生 |
| 圣骑 | 圣骑；命运系特殊职业，只用于萌萌等明确指定的特殊档案，不再映射为“今日勇士” |

不确定职业时，不要猜测后直接导入。先调用网站加载的职业资料库或 Supabase Edge Function 的 `listProfessions` 核对标准职业名，再生成文本。

职业对照表已按 16 位神明完整校验。近期修正的易错项：`受害者=欺诈刺客`、`驭兽师=欺诈猎人`、`魔术师=欺诈歌者`；`预言家=命运歌者`、`终末之笔=命运猎人`、`织命师=命运牧师`。

## 5. 导入操作

1. 粘贴带 8 列表头的 TSV 到“批量卷宗录入”。
2. 点击“检查内容”。
3. 肉眼核对每一行显示的 `昵称：职业，登神 X，觐见 Y`，确认没有串位。
4. 只有结果为 `有效行：N，错误：0` 时，点击“导入有效行”。
5. 若页面显示“后端校验发现错误，未导入”，整批没有写入；修正原文本后重新检查。
6. 导入成功后点击“刷新数据”，检查公开排行榜中的昵称、职业、信仰、命途和分数。
7. 使用一名新信徒的昵称与暗语进入私密面板，确认天赋与私密备注可见且未出现在公开榜单。

如果检查出现任何错误，整批不要提交。先修正错误行后再次检查。

## 6. 重名、改名与修复

### 同名重新导入

同昵称会更新档案，包括暗语、分数、公开短记、私密备注和天赋。不要把同名导入当作“只改一项”的工具。

### 改名

批量导入不能安全完成改名：使用新昵称会创建新档案。已存在档案改名要在 Supabase SQL Editor 执行单独 SQL，例如：

```sql
UPDATE public.hope_profiles
SET display_name = '新昵称'
WHERE display_name = '旧昵称'
RETURNING id, display_name, profession, faith_god, ascension_score, audience_score;
```

确认 `RETURNING` 只返回一行后，再刷新网站。

## 7. 分数结算与初始分数的区别

- 批量卷宗录入中的“登神分 / 觐见分”是档案初始或覆写后的绝对值。
- “群体结算”才使用增减分，格式为：`信仰 职业 昵称 +登神变化 +觐见变化`。
- 结算范围：登神分 `-20` 到 `+20`，觐见分 `-3` 到 `+3`。
- 不要用分数结算去补建没有档案的信徒；必须先导入档案。

## 8. 新线程执行清单

1. 先读取本说明书和当前 `hope-state-site/app.js` 的批量解析规则。
2. 只读 Excel，列出“导入昵称、标准职业、信仰/命途、两项初始分数”的核对表给用户确认。
3. 生成含固定 8 列表头的 TSV；保留暗语、私密备注、天赋；每个单元格必须单行且不得含制表符。
4. 在交付前逐行列出：昵称、标准职业、登神分、觐见分，供用户肉眼确认。
5. 将 TSV 放进用户剪贴板，不要自动调用 Supabase 写入。
6. 用户在网站中先检查内容，逐行核对职业与两项分数，再自行点击导入。
7. 若发现任意错误，整批重做；不要尝试只导入“看起来没问题的行”。
8. 遇到改名，提供单独 SQL，不要通过重新导入创建重复档案。

## 9. 部署前置条件

前端严格 TSV 检查依赖最新 GitHub Pages 页面。后端原子校验依赖最新 Supabase Edge Function。录入前确认：

1. GitHub Pages 已刷新到 `app.js?v=20260719-bulk-tsv-guard-1` 或之后版本。
2. Supabase 的 `hope-state-action` 已部署仓库中的最新 `supabase/functions/hope-state-action/index.ts`。
3. 本说明书对应的职业对照 SQL 已执行完成，尤其是欺诈与命运职业映射修复。
